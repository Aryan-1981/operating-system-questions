1Ô∏è‚É£ What is a Process? Program vs Process

üìå What is a Process?

Simple Definition: A process is a program that is currently running on your computer.

Think of it as a "living" version of your program - it has memory, it's doing work, and the operating system is keeping track of it.

üîÑ Program vs Process - The Key Difference

üíæ PROGRAM

  • Passive entity (not doing anything)
  • Stored on your hard disk
  • A file containing instructions
  • Like a recipe in a cookbook
  • Example: Chrome.exe on your computer

‚ö° PROCESS

  • Active entity (actively running)
  • Loaded in RAM (memory)
  • Program in execution
  • Like cooking the recipe
  • Example: Chrome browser running

üç≥ Real-Life Example:

Program: A recipe for making a cake (just written instructions)

Process: You actively making the cake using that recipe (mixing ingredients, baking, etc.)


Computer Example:

Program: Microsoft Word application file sitting on your hard drive

Process: When you double-click Word and it opens - now it's using CPU, memory, and doing work!

Aspect Program Process
Nature Static (doesn't change) Dynamic (changes as it runs)
Location Hard Disk/SSD RAM (Main Memory)
Lifespan Permanent (until deleted) Temporary (until terminated)
Resource Usage No resources needed Needs CPU, memory, I/O
Example notepad.exe Notepad running with your essay

üìù Detailed Explanation of Each State

1. NEW:

The process is being created. Like a baby being born - it exists but isn't ready to do anything yet.

2. READY:

Process is ready to run and waiting for CPU time. Like students raising their hands, waiting for the teacher to call on them.

3. RUNNING:

Process is currently executing on the CPU. Like the student who is speaking - they have the "CPU's attention."

4. WAITING/BLOCKED:

Process is waiting for some event (like input/output). Like waiting for a file to download - can't continue until it's done.

5. TERMINATED:

Process has finished execution. Like closing an app - it's done and removed from memory.

üéÆ Real Example: Playing a Video Game

  1. NEW: You double-click the game icon - it's loading
  2. READY: Game is loaded, waiting for CPU to start running it
  3. RUNNING: You're playing the game actively
  4. WAITING: Game is loading the next level from disk (waiting for I/O)
  5. RUNNING: Back to playing after level loads
  6. TERMINATED: You close the game

3Ô∏è‚É£ Important Process Concepts

(i) Process Control Block (PCB)

üìå What is PCB?

Simple Definition: PCB is like an ID card for each process. It contains all important information about the process.

The operating system maintains a PCB for every process to keep track of it.

üìã Process Control Block Contents

Process Control Block (PCB)

Process ID: Unique number (like 1234)

Process State: New/Ready/Running/Waiting/Terminated

Program Counter: Next instruction to execute

CPU Registers: Current values of registers

Memory Limits: Where in memory this process lives

Open Files: List of files this process is using

CPU Scheduling Info: Priority, queue pointers

I/O Status: Devices allocated to process

üè´ Student ID Card Analogy:

Just like your student ID has:

  • Your name (Process ID)
  • Your photo (Process state)
  • Your courses (Resources/files)
  • Your semester (Priority)

PCB has all details about a process!

(ii) Context Switching

üìå What is Context Switching?

Simple Definition: Context switching is when the CPU stops running one process and starts running another process.

It's like switching between apps on your phone!

üîÑ Context Switch Process

üìù Steps in Context Switching:

  1. Save the state of the currently running process (P1) to its PCB
  2. Select the next process (P2) to run (by the scheduler)
  3. Load the state of P2 from its PCB
  4. Resume execution of P2
‚ö†Ô∏è Important: Context switching takes time (called overhead). During this time, the CPU is NOT doing useful work - it's just switching processes. That's why we want to minimize context switches!

üì± Smartphone Analogy:

Imagine you're using your phone:

  1. You're on Instagram (Process P1 running)
  2. A message arrives, you pause Instagram (save P1 state)
  3. You switch to WhatsApp (load P2 state)
  4. You read the message (Process P2 running)
  5. You switch back to Instagram - it resumes where you left off!

That's context switching!

(iii) Process Scheduling

üìå What is Process Scheduling?

Simple Definition: Process scheduling is the method by which the operating system decides which process gets to use the CPU and for how long.

Think of it as a teacher deciding which student gets to answer next!

üéØ Types of Scheduling Queues:

Job Queue:

All processes in the system

Ready Queue:

Processes in memory, ready and waiting to execute

Device Queue:

Processes waiting for I/O devices

üìä Types of Schedulers:

Scheduler Type What it Does Speed
Long-term Scheduler (Job Scheduler) Decides which processes to load into memory Slow (minutes)
Short-term Scheduler (CPU Scheduler) Decides which ready process gets CPU next Very Fast (milliseconds)
Medium-term Scheduler Swaps processes in and out of memory Medium speed

üè™ Restaurant Analogy:

  • Long-term Scheduler: The host deciding which customers to seat (from waiting list to table)
  • Short-term Scheduler: The waiter deciding which seated customer to serve next
  • Medium-term Scheduler: Moving customers to the bar while waiting for their table

4Ô∏è‚É£ CPU Scheduling Criteria

üìå What are Scheduling Criteria?

These are the metrics (measurements) we use to judge how good a scheduling algorithm is. Like grades for evaluating student performance!

üìä Main Scheduling Criteria:

1. CPU Utilization

Goal: Keep CPU as busy as possible

Range: 0% to 100% (we want HIGH)

Analogy: Like keeping a worker busy all day, not idle

2. Throughput

Goal: Number of processes completed per time unit

Measure: Processes/hour (we want HIGH)

Analogy: How many customers a cashier serves per hour

3. Turnaround Time

Goal: Time from submission to completion

Formula: Completion Time - Arrival Time (we want LOW)

Analogy: Time from ordering pizza to receiving it

4. Waiting Time

Goal: Time spent waiting in the ready queue

Measure: Total waiting time (we want LOW)

Analogy: Time spent standing in line at Starbucks

5. Response Time

Goal: Time from submission until first response

Measure: Time to first output (we want LOW)

Analogy: Time until your video starts buffering after clicking play

üìê Important Formulas:

Turnaround Time = Completion Time - Arrival Time
Waiting Time = Turnaround Time - Burst Time
Response Time = First CPU Time - Arrival Time

üéØ Optimization Goals:

  • Maximize: CPU Utilization, Throughput
  • Minimize: Turnaround Time, Waiting Time, Response Time

5Ô∏è‚É£ CPU Scheduling Algorithms

Algorithm 1: FCFS (First-Come, First-Served)

üìå What is FCFS?

Simple Definition: The process that arrives first gets the CPU first. Like standing in a line - first person in line gets served first!

Type: Non-preemptive (once a process starts, it runs until completion)

‚úÖ Advantages:

  • Very simple to understand and implement
  • Fair - everyone gets served in order
  • No starvation - everyone eventually runs

‚ùå Disadvantages:

  • Convoy Effect: Short processes wait for long ones
  • Poor average waiting time
  • Not suitable for time-sharing systems

Algorithm 2: SJF (Shortest Job First)

üìå What is SJF?

Simple Definition: The process with the shortest CPU burst time gets executed first. Like letting people with fewer items go first at checkout!

Type: Can be preemptive (SRTF - Shortest Remaining Time First) or non-preemptive

‚úÖ Advantages:

  • Optimal - gives minimum average waiting time
  • Better throughput than FCFS
  • Short processes finish quickly

‚ùå Disadvantages:

  • Starvation: Long processes may never execute!
  • Difficult to predict burst time in advance
  • Not practical for short-term scheduling

Algorithm 3: Round Robin (RR)

üìå What is Round Robin?

Simple Definition: Each process gets a small time slice (quantum) of CPU time. If it doesn't finish, it goes to the back of the line. Like taking turns on a playground slide!

Type: Preemptive

Key Parameter: Time Quantum (usually 10-100 milliseconds)

‚öôÔ∏è Time Quantum Selection:

  • Too Large: Becomes like FCFS (defeats the purpose)
  • Too Small: Too many context switches (high overhead)
  • Rule of Thumb: 80% of CPU bursts should be shorter than time quantum

‚úÖ Advantages:

  • Fair - every process gets equal CPU time
  • No starvation - everyone gets turns
  • Best for time-sharing systems
  • Good response time

‚ùå Disadvantages:

  • Higher average turnaround time than SJF
  • Performance depends on time quantum
  • More context switches = more overhead

üèÜ Which Algorithm is Best for Interactive Systems?

Answer: Round Robin is BEST for Interactive Systems!

Why?

  • ‚úì Excellent Response Time: Users get feedback quickly
  • ‚úì Fair Time Sharing: Every process gets regular CPU time
  • ‚úì No Starvation: Every process eventually runs
  • ‚úì Predictable: Users know they'll get response soon

üíª Interactive System Examples:

  • Text Editors: Need quick response when you type
  • Web Browsers: Need to respond to clicks quickly
  • Games: Need smooth, responsive gameplay
  • Operating System UI: Windows, icons need to respond instantly

Round Robin ensures all these programs get regular CPU time, so none feel "frozen"!
